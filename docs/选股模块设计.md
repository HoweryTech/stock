# 选股模块设计

本文档定义个人 A 股投资决策系统中的选股模块边界、当前实现状态和后续扩展顺序。

选股模块的目标不是直接给出买入指令，而是生成可解释、可复盘、可进入交易计划门禁的观察池。真实交易仍必须由用户补全买入价、止损价、仓位和失效条件，并通过风控校验。

## 1. 决策边界

选股模块只回答：

- 哪些股票符合当前投资体系下的观察条件。
- 入选依据来自哪些策略和证据。
- 还缺少哪些反证、风险、流动性、行业或组合上下文。
- 是否具备进入交易计划草稿的最低证据完整性。

选股模块不回答：

- 是否应该立即买入。
- 应该买入多少。
- 是否绕过止损、仓位或组合约束。
- 是否自动下单。

## 2. 当前已实现链路

当前第一版链路以本地标准化 CSV 为输入：

```text
stock_universe.csv
  -> tradable_universe.csv

daily_bars.csv
  -> trend_factors.csv
  -> trend_candidates.csv
  -> industry_strength_factors.csv

financial_metrics.csv + valuation_metrics.csv
  -> value_quality_candidates.csv

event_catalyst_events.csv
  -> event_catalyst_candidates.csv

trend_candidates.csv + value_quality_candidates.csv + event_catalyst_candidates.csv
  -> candidate_pool.csv
  -> candidate_pool_check
  -> reports/watchlist.md
```

对应工具：

- `tools/filter_universe.py`：按投资体系过滤可交易股票池。
- `tools/calc_trend_factors.py`：计算基础趋势因子。
- `tools/calc_industry_strength.py`：按行业聚合收益、成交额和上涨占比，生成行业强度因子。
- `tools/screen_trend_strength.py`：筛选趋势强度候选。
- `tools/screen_value_quality.py`：筛选价值质量候选。
- `tools/screen_event_catalyst.py`：筛选结构化事件催化候选。
- `tools/merge_candidate_pool.py`：合并多策略候选池。
- `tools/check_candidate_pool.py`：检查候选池证据门禁。
- `tools/generate_watchlist_report.py`：生成观察池报告。
- `tools/track_candidate_performance.py`：跟踪候选入池后的 5/10/20 日表现。
- `tools/prepare_trade_plan_from_candidate.py`：从候选进入交易计划门禁。

## 3. 股票池分层

选股模块按以下层级推进：

1. 全市场股票池：基础股票信息和行业分类。
2. 可交易股票池：排除 ST、退市风险、停牌、异常交易、上市天数不足、成交额过低等标的。
3. 风险过滤股票池：排除重大未处理公告、财务、监管或诚信风险。
4. 策略候选池：趋势强度、价值质量、事件催化、行业强度等策略独立产出候选。
5. 今日重点观察池：多策略合并、风险扣分、组合约束和人工复核后的观察列表。

当前代码已覆盖第 1、2、4、5 层的一部分；第 3 层和行业/事件/组合适配仍需扩展。

当前候选池的真实运行状态需要和设计目标分开看：如果 `value_quality_candidates.csv` 或 `event_catalyst_candidates.csv` 为空，则候选虽然进入统一候选池，但 `strategies` 会全部显示为 `trend_strength`。这表示候选只通过了短期趋势强度筛选，没有获得财务质量或事件催化的交叉验证。页面和报告应把这种候选理解为“趋势观察”，不能当成完整多因子推荐。

## 4. 候选输出结构

统一候选池至少保留：

- `code`：股票代码。
- `strategies`：命中的策略来源。
- `primary_strategy`：主策略或 `multi_strategy`。
- `combined_score`：观察池排序分。
- `strategy_confluence_score`：策略共振分。
- `strategy_confluence_evidence`：策略共振证据。
- `trend_score`：趋势强度分。
- `value_quality_score`：价值质量分。
- `event_score`：事件催化分。
- `event_date`：事件日期。
- `event_type`：事件类型。
- `name`：股票名称。
- `industry`：所属行业。
- `exchange`：交易所，如 `SSE`、`SZSE`、`BSE`。
- `board`：板块，如沪市主板、深市主板、科创板、创业板、北交所。
- `liquidity_score`：流动性评分。
- `liquidity_evidence`：流动性证据。
- `data_quality_score`：数据质量分。
- `data_quality_status`：数据质量状态。
- `data_quality_evidence`：数据质量证据。
- `risk_penalty_score`：风险扣分。
- `risk_penalty_evidence`：风险扣分证据。
- `technical_health_score`：日线、周线、月线技术健康分。
- `technical_health_status`：技术健康状态，取值为 `strong`、`watch`、`weak`、`blocked` 或 `insufficient`。
- `technical_health_evidence`：MACD、BOLL、RSI、KDJ、ATR、量比等技术证据摘要。
- `technical_risk_flags`：技术风险标记，如 MACD 死叉、BOLL 下轨区、量比不足等。
- `portfolio_fit_status`：组合适配状态。
- `portfolio_fit_action`：组合适配动作。
- `portfolio_fit_evidence`：组合适配证据。
- `trade_date`：趋势证据交易日。
- `report_period`：财务证据报告期。
- `reasons`：带策略前缀的入选证据。
- `risks`：带策略前缀的风险和反证。

## 5. 指标体系

### 5.1 硬过滤指标

- ST 或退市风险。
- 停牌或异常交易状态。
- 上市天数。
- 近 20 日平均成交额。
- 涨跌停状态。
- 数据是否缺失或过旧。

硬过滤不应被策略分数抵消。

### 5.2 基本面质量指标

- ROE、ROA。
- 毛利率、净利率。
- 经营现金流净额。
- 经营现金流与净利润匹配度。
- 资产负债率和有息负债。
- 营收同比。
- 净利润同比和扣非净利润同比。
- 财报异常、监管问询和财务更正。

### 5.3 估值指标

- PE TTM、PB、PS。
- PE/PB 历史分位。
- 行业估值分位。
- 股息率。
- PEG 或盈利增速与估值匹配度。

估值指标用于判断安全边际，不单独构成买入理由。当前 PE/PB 分位先使用全市场横截面正值样本派生，行业估值分位后续从历史估值快照中月频重算。

### 5.4 技术趋势指标

- 近 5/20/60 日收益率。
- MA5/10/20/60 结构。
- 是否站上关键均线。
- 日线、周线、月线 MACD DIF/DEA、柱体和交叉状态。
- BOLL 中轨、上下轨、%b 和带宽。
- RSI6/14 的强弱、超买和超卖状态。
- KDJ K/D/J 的方向、钝化和超买风险。
- ATR14 波动率，用于判断止损距离和仓位风险。
- 最新量比相对 20 根均量的确认或背离。
- 相对大盘强度。
- 相对行业强度。
- 突破、回踩和支撑位。
- 成交额变化。
- 波动率和最大回撤。

技术趋势指标在候选池中按门禁/扣分型因子使用：多周期共振健康只小幅加分；日线 MACD 死叉、日线转弱叠加周线转弱、跌至 BOLL 弱区、RSI/KDJ 偏弱、量比不足或 ATR 过高会扣分。`technical_health_status=blocked` 的候选会被额外下调综合推荐度，不能直接进入买入计划，只能保留为人工观察或排除。

### 5.5 行业与市场环境指标

- 行业指数近 5/20/60 日收益。
- 行业相对大盘强度。
- 行业内上涨家数占比。
- 行业成交额变化。
- 市场宽度。
- 弱市/强市仓位规则。

行业强度应作为下一阶段优先补齐的因子。

### 5.6 资金与交易活跃度指标

- 主力资金净流入。
- 大单净额。
- 换手率。
- 融资融券变化。
- 北向资金持股变化。
- 龙虎榜和大宗交易。

资金指标适合作为确认或风险提示，不宜单独成为买入理由。

### 5.7 事件催化指标

- 业绩预告。
- 回购。
- 增持或减持。
- 解禁。
- 重大合同。
- 政策催化。
- 监管问询。
- 重组和并购。

事件候选必须同时输出正向催化、反证、风险和失效条件。

### 5.8 组合适配指标

- 当前单票仓位。
- 所属行业已有仓位。
- 买入后总仓位。
- 现金比例。
- 单笔最大亏损。
- 同风格或同主题暴露。
- 是否处于亏损冷静期。

组合适配不改变股票本身质量，也不改变 `combined_score`，但决定候选是可准备交易计划、继续观察，还是因组合约束暂缓。

### 5.9 页面筛选指标

页面选股列表应至少支持：

- 按交易所筛选：上证、深证、北交所。
- 按板块筛选：沪市主板、深市主板、科创板、创业板、北交所。
- 按行业、策略来源、组合适配状态、数据质量状态和技术健康状态筛选。
- 按综合分、策略数量、行业强度、流动性、风险扣分、技术健康分和买入后总仓位排序。

### 5.10 页面人工选股规则

页面上的 `combined_score` 是观察池排序入口，不是买入结论。人工查看候选列表时按以下顺序处理：

1. 先过滤无权限板块：当前默认排除科创板、创业板和北交所，只看沪市主板和深市主板。
2. 先看技术门禁：`technical_health_status=blocked` 直接排除，不进入买入计划；`weak` 只保留观察；`watch` 可以进入重点观察；`strong` 优先级最高。
3. 再看综合分：在权限和技术门禁通过后，按 `combined_score` 从高到低查看；综合分高但技术面 blocked 的候选不能因为分高而绕过门禁。
4. 再看策略来源：多策略共振优先于单策略来源。`trend_strength` 只代表短期趋势观察；`value_quality` 代表财务质量确认；`event_catalyst` 代表公告或事件输入，但仍需要复核事件持续性和反证。
5. 再看技术证据：重点检查日线和周线 MACD 是否转弱或死叉、RSI 是否过热或过弱、KDJ 是否钝化或下行、BOLL 位置是否追高或跌破中轨、ATR 是否过高、量比是否支持上涨。
6. 再看流动性：优先选择成交额充足、`liquidity_score` 较高、没有停牌或极端缩量风险的候选；流动性差的候选即使综合分高也只进入观察。
7. 最后看风险提示：若 `risks`、`risk_penalty_evidence` 或事件字段提示监管、减持、解禁、诉讼、财务更正、未提供显式风险等问题，必须先人工复核，不能直接生成交易计划。

默认人工查看口径：

- 优先：主板、综合分靠前、`technical_health_status` 为 `strong` 或 `watch`、流动性好、多策略来源。
- 排除：无权限板块、`technical_health_status=blocked`、风险事件、流动性差。
- 仅观察：综合分高但技术状态为 `weak`、单纯趋势票、事件只有标题初筛且缺少进一步验证。

页面默认行为应和该口径一致：选股页默认开启“重点观察”，排除科创板、创业板和北交所，并只展示 `technical_health_status` 为 `strong` 或 `watch` 的候选；需要排查完整候选池时，再手动关闭“重点观察”或调整技术面筛选。

## 6. 数据接入优先级

选股模块的数据接入按“先恢复多策略有效性，再提高解释完整性”的顺序推进。

### 6.1 第一优先级：财务质量指标

目标是让 `value_quality` 策略从 0 候选恢复为可用策略。需要接入：

- ROE、ROA。
- 毛利率、净利率。
- 经营现金流净额。
- 资产负债率。
- 营收同比。
- 净利润同比和扣非净利润同比。
- EPS 和报告期。

首次建库拉取近 3-5 年报告期；后续按 `code + report_period` 增量更新。财报季可以每日补拉最近报告期，平时季度刷新即可。原始抓取结果必须保留到 `data/raw/snapshots/financial_metrics`，标准化结果写入 `data/processed/financial_metrics.csv`。

价值质量评分需要对极端同比增速做贡献封顶。原始同比仍保留在入选证据里，但不能让低基数造成的几千个百分点增长直接支配综合排序。

### 6.2 第二优先级：公告和事件催化

目标是让 `event_catalyst` 策略产生可复核候选，同时补齐风险公告过滤。需要接入：

- 公告 ID、公告日期、标题、股票代码。
- 公告分类或栏目。
- 事件类型，如业绩预告、回购、增持、重大订单、并购重组、政策催化。
- 风险事件，如监管问询、立案调查、减持、质押冻结、诉讼、退市风险。
- 影响评分、可信度、反证检查和风险披露。

首次建库拉取近 6-12 个月公告；后续每天拉最近 1-7 天公告，并按公告 ID 去重。原始公告保留到 `data/raw/snapshots/event_catalyst_events`，结构化事件写入 `data/processed/event_catalyst_events.csv`。事件策略不能只凭公告标题直接推荐，必须保留人工复核证据和反证字段。

事件输入分两步落地：

1. `tools/fetch_eastmoney_announcements.py` 负责抓取公告、按公告 ID 去重、留存快照，并用保守标题规则生成结构化事件输入。
2. `tools/screen_event_catalyst.py` 只从结构化事件中筛出正向且证据完整的催化候选；减持、问询、处罚、诉讼、财务更正、解禁等风险事件只进入风险复核，不生成事件催化推荐。

初始标题规则只作为分流和待复核标签，不等同于最终基本面判断。事件字段里的 `expected_impact`、`counter_evidence` 和 `risk_disclosure` 必须保留，避免把公告标题直接当作买入理由。

### 6.3 第三优先级：估值补强和行业估值分位

当前 PE TTM、PB、市值和流通市值已经可用；PS、PCF、股息率和行业估值分位仍需补齐可靠来源或从历史快照派生。

估值快照适合周频刷新或市场剧烈波动时手动补跑；行业估值分位适合月频重算。后续入库时按 `code + trade_date` 保存估值快照，按 `industry + metric + as_of_date` 保存行业分位。

### 6.4 数据留存和未来入库原则

所有新接入数据都按三层保存：

1. `data/raw/snapshots`：每次抓取的原始结果，按日期和时间留存，不覆盖。
2. `data/processed`：当前可用标准化宽表，供策略流水线读取，可以由原始快照重建。
3. `data/metadata`：抓取时间、来源、字段映射、行数、失败记录和质量检查结果。

后续数据库以这些唯一键做 upsert：行情用 `code + trade_date`，估值用 `code + trade_date`，财务用 `code + report_period`，公告用 `announcement_id`，候选池快照用 `trade_date + code + strategy_version`。

## 7. 质量门禁

候选进入交易计划前，至少需要满足：

- 有股票代码、策略来源、主策略和入选证据。
- 趋势候选有交易日和趋势分。
- 价值质量候选有报告期和价值质量分。
- 如传入可交易股票池，候选必须仍在可交易股票池中。
- 至少具备两个非风险证据维度，或降级为人工复核。
- 有显式风险提示，否则降级为人工复核。
- 有流动性证据，否则降级为人工复核。

`blocked` 不应进入交易计划草稿；`needs_review` 可以进入人工研究，但不能直接视为可交易。

## 8. 后续实施顺序

建议按以下顺序迭代：

1. 固化当前设计文档和候选池质量门禁。
2. 为候选池补股票名称、行业和流动性评分。
3. 增加行业强度因子和行业相对强弱排序。
4. 增加事件催化候选文件并接入统一候选池。
5. 已重构综合评分为策略共振、风险扣分、流动性、行业强度和数据质量共同决定；市场环境可在后续作为独立上下文接入。
6. 已接入组合适配，输出观察、可准备计划、因组合约束暂缓。
7. 已将候选入池后的 5/10/20 日表现汇总接入策略健康度的候选观察区，后续可继续细化策略共振和组合适配。
