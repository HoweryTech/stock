# 每日决策流程

本文档定义第一阶段从数据准备到交易计划准入门禁、持仓检查、卖出纪律和复盘草稿的标准操作顺序。

目标是让每天的选股和计划生成可验证、可复盘、可控制风险，而不是临时凭感觉交易。

## 1. 数据准备

真实数据不提交到 Git。每天先把外部数据转换为本项目标准 CSV。

```bash
python3 tools/import_stock_universe.py \
  --input data/raw/stock_universe.csv \
  --output data/processed/stock_universe.csv \
  --metadata-output data/metadata/stock_universe.import.json

python3 tools/filter_universe.py \
  --profile config/investment-profile.example.yaml \
  --input data/processed/stock_universe.csv \
  --output data/processed/tradable_universe.csv \
  --report-output data/metadata/tradable_universe.filter.json

python3 tools/import_daily_bars.py \
  --input data/raw/daily_bars.csv \
  --output data/processed/daily_bars.csv \
  --metadata-output data/metadata/daily_bars.import.json

python3 tools/import_financial_metrics.py \
  --input data/raw/financial_metrics.csv \
  --output data/processed/financial_metrics.csv \
  --metadata-output data/metadata/financial_metrics.import.json

python3 tools/import_valuation_metrics.py \
  --input data/raw/valuation_metrics.csv \
  --output data/processed/valuation_metrics.csv \
  --metadata-output data/metadata/valuation_metrics.import.json
```

## 2. 生成观察池

```bash
python3 tools/run_watchlist_pipeline.py \
  --daily-bars data/processed/daily_bars.csv \
  --financial-metrics data/processed/financial_metrics.csv \
  --valuation-metrics data/processed/valuation_metrics.csv \
  --universe data/processed/tradable_universe.csv \
  --report-output reports/watchlist.md \
  --metadata-output data/metadata/watchlist_pipeline.json
```

流水线会输出：

- 趋势强度因子。
- 趋势强度候选。
- 价值质量候选。
- 多策略候选池。
- 候选池质量检查结论。
- 观察池报告。

生成观察池和组合检查后，可以随时生成每日操作摘要：

```bash
python3 tools/generate_daily_summary.py \
  --output reports/daily-summary.md \
  --json-output data/metadata/daily-summary.json
```

每日操作摘要会读取策略健康检查结果、策略复核任务和策略配置变更草稿，并列出暂停新开仓或需复核策略的具体原因；如果存在亏损纪律例外交易，会在“策略健康动作”中显示对应策略和复核原因。若存在未完成/暂缓的策略复核任务，或待审批的配置变更草稿，也会进入今日优先动作。

## 3. 检查候选池

如果流水线元数据中的 `steps.candidate_pool_check.conclusion` 不是 `pass`，需要先阅读提醒。

也可以单独执行：

```bash
python3 tools/check_candidate_pool.py \
  --candidates data/processed/candidate_pool.csv
```

原则：

- `blocked`：不进入交易计划。
- `needs_review`：可以观察，但生成交易计划前需要补齐提醒项。
- `pass`：字段完整性通过，仍不代表可以买入。

## 4. 从候选股准备交易计划

```bash
python3 tools/prepare_trade_plan_from_candidate.py \
  --candidates data/processed/candidate_pool.csv \
  --code 300750 \
  --name 宁德时代 \
  --exchange SZSE \
  --industry 电力设备 \
  --strategy-config-snapshot data/metadata/strategy-config-snapshot.json \
  --planned-buy-price 200 \
  --stop-loss-price 185 \
  --position-pct 5 \
  --stop-loss-condition "收盘价跌破 185。" \
  --take-profit-condition "达到计划目标区后分批止盈。" \
  --invalidation-condition "趋势或估值证据失效。" \
  --review-focus "候选池证据是否被市场验证。" \
  --mark-ready \
  --gate-output data/metadata/TP-300750.gate.json
```

该命令会：

1. 读取候选池证据。
2. 生成交易计划草稿。
3. 补齐人工交易条件。
4. 重新计算衍生风控字段。
5. 记录策略配置版本快照。
6. 执行交易计划准入门禁。

如果当天刚完成策略配置变更，先执行第 23 步生成新的策略配置版本快照，再生成新交易计划。

## 5. 准入门禁结论

门禁结论处理方式：

- `blocked_by_quality`：计划草稿没写完整，不能看作风控结果。
- `blocked_by_risk`：计划写完整，但违反价格、止损、仓位等风控规则。
- `needs_confirmation`：没有阻断项，但仍需人工确认。
- `pass`：可进入后续执行记录流程。

## 6. 生成执行记录

门禁结论为 `needs_confirmation` 或 `pass` 后，仍不直接生成持仓，先生成执行记录。

模拟执行：

```bash
python3 tools/new_trade_execution.py \
  --plan plans/TP-300750.yaml \
  --gate data/metadata/TP-300750.gate.json \
  --strategy-config-snapshot data/metadata/strategy-config-snapshot.json \
  --mode paper \
  --execution-date 2026-07-07 \
  --execution-price 200 \
  --shares 100 \
  --position-pct 5 \
  --user-confirmed \
  --note "按计划模拟成交。"
```

真实执行时使用 `--mode real`，并且必须传入 `--user-confirmed`。

## 7. 生成持仓记录

成交后从执行记录生成持仓：

```bash
python3 tools/new_position_from_execution.py \
  --execution executions/EXEC-示例.yaml \
  --current-price 200 \
  --days-held 0 \
  --note "从执行记录生成持仓。"
```

持仓记录会保留执行记录快照和交易计划快照。
如果执行偏差检查为 `blocked`，默认不会生成持仓。

## 8. 交易前最后要求

真实交易前必须确认：

- 计划文件已保存。
- 门禁 JSON 已保存。
- 执行记录会保存。
- 交易计划中没有占位内容。
- 止损价、仓位、最大亏损和最大可接受买入价已确认。
- 反证和风险已阅读。
- 接受按止损价计算的最大亏损。

没有通过准入门禁，不进入交易。

## 9. 持仓后每日检查

生成持仓后，每个交易日更新当前价并执行持仓日检：

```bash
python3 tools/update_position_daily.py \
  --position positions/POS-示例.yaml \
  --current-price 201 \
  --days-held 1 \
  --note "成交额继续放大，趋势仍在。" \
  --check-output data/metadata/POS-示例.daily-check.json
```

如果日检结论为 `needs_action`，必须先处理止损、仓位或其他风险，不继续加仓。

## 10. 组合持仓检查

完成单个持仓日检后，执行组合级汇总：

```bash
python3 tools/check_portfolio_positions.py \
  --positions positions/*.yaml \
  --output data/metadata/portfolio_positions.check.json
```

组合检查用于确认：

- 总仓位是否接近或超过上限。
- 行业仓位是否接近或超过上限。
- 是否存在需要处理的持仓。
- 是否存在提醒级持仓。

## 11. 生成退出计划

当持仓日检出现 `needs_action`，或主观上准备止盈、降风险、验证买入假设失效时，先生成退出计划，不直接卖出。

```bash
python3 tools/new_exit_plan.py \
  --position positions/POS-示例.yaml \
  --daily-check data/metadata/POS-示例.daily-check.json \
  --output exit-plans/EXIT-示例.yaml
```

退出计划必须明确：

- 为什么退出。
- 退出多少仓位。
- 计划退出价格和最低可接受退出价。
- 是否符合原交易计划。
- 继续持有的风险。

## 12. 检查退出计划

```bash
python3 tools/check_exit_plan.py \
  --exit-plan exit-plans/EXIT-示例.yaml
```

结论处理方式：

- `blocked`：不能进入卖出执行，先修正退出计划。
- `needs_review`：可以卖出，但必须人工确认并保留原因。
- `pass`：可以进入卖出执行记录。

## 13. 生成卖出执行记录

退出计划检查通过后，生成卖出执行记录：

```bash
python3 tools/new_exit_execution.py \
  --exit-plan exit-plans/EXIT-示例.yaml \
  --execution-date 2026-07-08 \
  --execution-price 9.1 \
  --shares 1000 \
  --output exit-executions/EXITEXEC-示例.yaml
```

真实卖出必须传入 `--mode real --user-confirmed`。

## 14. 生成交易复盘草稿

卖出执行记录生成后，立即生成复盘草稿：

```bash
python3 tools/new_trade_review_from_exit_execution.py \
  --exit-execution exit-executions/EXITEXEC-示例.yaml \
  --output reviews/TR-示例.yaml
```

复盘草稿会自动带入：

- 来源交易计划。
- 来源持仓。
- 来源退出计划。
- 来源卖出执行。
- 入场价、退出价、退出仓位和收益贡献。
- 入场时是否属于冷静期或策略暂停期例外交易。

复盘结论、经验教训和下一步动作仍需要人工补充。

## 15. 复盘维护流水线

复盘补全后，执行复盘维护流水线，更新策略统计和冷静期状态：

```bash
python3 tools/run_review_pipeline.py \
  --reviews reviews/*.yaml
```

该命令会生成：

- `reports/review-analysis.md`
- `reports/strategy-health.md`
- `reports/strategy-review-tasks.md`
- `data/metadata/review-analysis.json`
- `data/metadata/review-cooldown.json`
- `data/metadata/strategy-health.json`
- `data/metadata/strategy-review-tasks.json`
- `data/metadata/review-pipeline.json`

如果冷静期结论为 `cooldown_required`，后续买入执行默认阻断。

## 16. 生成策略复核任务

复盘维护流水线会自动生成策略复核任务清单。也可以单独重新生成：

```bash
python3 tools/generate_strategy_review_tasks.py \
  --strategy-health data/metadata/strategy-health.json \
  --output reports/strategy-review-tasks.md \
  --json-output data/metadata/strategy-review-tasks.json
```

策略复核任务必须形成明确结论：暂停新开仓、继续观察、调整规则或降低仓位上限。没有完成复核前，不扩大对应策略的风险暴露。

## 17. 生成策略配置变更草稿

策略复核任务标记为 `resolved` 后，可以生成策略配置变更草稿：

```bash
python3 tools/generate_strategy_config_changes.py \
  --tasks data/metadata/strategy-review-tasks.json \
  --output reports/strategy-config-changes.md \
  --json-output data/metadata/strategy-config-changes.json
```

该命令不直接修改投资体系配置。真正修改 `config/investment-profile.example.yaml` 前，必须确认来源任务、复盘证据、变更字段、审批人、审批理由和生效日期。配置版本变更还必须保留 `config_version_id`、`profile_hash` 和人工复核结论。

## 18. 策略配置变更应用前校验

配置变更草稿审批后，先执行应用前校验：

```bash
python3 tools/check_strategy_config_changes.py \
  --profile config/investment-profile.example.yaml \
  --changes data/metadata/strategy-config-changes.json \
  --output data/metadata/strategy-config-changes.check.json
```

校验会阻断未审批变更、缺少审批人或审批时间、配置版本变更缺少审批理由或复核证据、无效配置路径、类型不匹配和超出安全边界的风险参数。只有结论为 `pass` 时，才允许进入生成待应用配置补丁。

## 19. 生成待应用配置补丁

配置变更校验通过后，生成待应用补丁：

```bash
python3 tools/generate_strategy_config_patch.py \
  --profile config/investment-profile.example.yaml \
  --changes data/metadata/strategy-config-changes.json \
  --check data/metadata/strategy-config-changes.check.json \
  --output reports/strategy-config-patch.md \
  --json-output data/metadata/strategy-config-patch.json
```

补丁只列出旧值、新值和来源证据，仍不自动修改真实配置文件。

## 20. 应用配置补丁

人工复核补丁无误后，应用配置补丁：

```bash
python3 tools/apply_strategy_config_patch.py \
  --profile config/investment-profile.example.yaml \
  --patch data/metadata/strategy-config-patch.json \
  --backup-dir data/backups \
  --audit-output data/metadata/strategy-config-patch.apply.json \
  --applied-by lihongwei
```

应用前会校验当前配置值必须等于补丁中的旧值，并先备份原配置。应用后必须复查审计记录，再运行相关策略筛选和风控测试。

## 21. 配置应用后回归检查

配置补丁应用后，立即执行回归检查：

```bash
python3 tools/check_strategy_config_regression.py \
  --profile config/investment-profile.example.yaml \
  --audit data/metadata/strategy-config-patch.apply.json \
  --output data/metadata/strategy-config-regression.json
```

如果回归结论为 `blocked`，不继续生成新计划，先回滚或修复配置。只有回归通过后，策略规则变更才算闭环完成。

## 22. 配置变更闭环流水线

也可以使用流水线统一执行配置变更链路：

```bash
python3 tools/run_strategy_config_change_pipeline.py \
  --profile config/investment-profile.example.yaml \
  --changes data/metadata/strategy-config-changes.json \
  --metadata-output data/metadata/strategy-config-change-pipeline.json
```

默认只执行应用前校验和待应用补丁生成，不修改配置。

需要应用时必须显式确认：

```bash
python3 tools/run_strategy_config_change_pipeline.py \
  --profile config/investment-profile.example.yaml \
  --changes data/metadata/strategy-config-changes.json \
  --apply \
  --applied-by lihongwei \
  --metadata-output data/metadata/strategy-config-change-pipeline.json
```

## 23. 生成策略配置版本快照

配置变更流水线通过后，固化当前投资体系配置版本：

```bash
python3 tools/create_strategy_config_snapshot.py \
  --profile config/investment-profile.example.yaml \
  --pipeline data/metadata/strategy-config-change-pipeline.json \
  --audit data/metadata/strategy-config-patch.apply.json \
  --regression data/metadata/strategy-config-regression.json \
  --output reports/strategy-config-snapshot.md \
  --json-output data/metadata/strategy-config-snapshot.json
```

该步骤只记录版本事实，不修改配置，也不生成交易建议。若 `profile_hash` 变化，应把后续交易计划、执行记录、复盘和回测绑定到新的 `version_id` 或 `profile_hash`。
