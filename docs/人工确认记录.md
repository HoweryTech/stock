# 人工确认记录

本文档定义如何记录每日摘要中的必须人工确认事项。

人工确认记录不生成买卖建议，也不自动推进交易或配置应用。它只记录确认人、确认时间、确认结论和理由，避免关键风险事项停留在口头确认。

## 1. 默认输出

```text
data/metadata/manual-confirmations.json
```

真实确认记录默认不提交到 Git。

## 2. 状态

每条确认事项支持三种动作：

- `confirm`：确认通过，必须填写理由。
- `reject`：驳回，必须填写理由。
- `reopen`：重新打开，回到待确认状态。

## 3. 命令

```bash
python3 tools/update_manual_confirmation.py \
  --confirmations data/metadata/manual-confirmations.json \
  --confirmation-id CONFIRM-CONFIG-CHANGE-RISK \
  --action confirm \
  --actor lihongwei \
  --reason "已核对旧值、新值和来源证据。" \
  --subject-type config_change \
  --subject-id CONFIG-CHANGE-RISK \
  --text "人工复核待应用配置补丁。"
```

驳回：

```bash
python3 tools/update_manual_confirmation.py \
  --confirmation-id CONFIRM-CONFIG-CHANGE-RISK \
  --action reject \
  --actor lihongwei \
  --reason "证据不足，暂不继续。"
```

重新打开：

```bash
python3 tools/update_manual_confirmation.py \
  --confirmation-id CONFIRM-CONFIG-CHANGE-RISK \
  --action reopen \
  --actor lihongwei \
  --reason "新增证据需要复核。"
```

## 4. 记录内容

每条记录包含：

- `id`：稳定确认事项 ID。
- `subject_type`：事项类型，例如 `exit_plan`、`config_change`、`config_patch`。
- `subject_id`：关联对象 ID。
- `text`：确认事项文本。
- `status`：`open`、`confirmed` 或 `rejected`。
- `confirmed_by`、`confirmed_at`、`confirmation_reason`。
- `rejected_by`、`rejected_at`、`rejected_reason`。
- `history`：每次状态变化的审计记录。
